# 0.11.0 Search Audit and Validation

`0.11.0` expands the search lane beyond basic query execution. The release now includes audit capture, backend health visibility, and explicit Elasticsearch validation hooks that make search behavior more reviewable and safer to operate.

## What shipped

- search queries are captured after server-side execution, subject to platform privacy controls
- captured queries can be disabled entirely or stored in hashed form
- the search backend is registry-based and explicitly uses Elasticsearch on this branch
- live validation and audit-oriented testing cover the real Elasticsearch path instead of relying only on stubbed search behavior

## Key implementation points

- [SearchQueryCaptureService](../../app/services/better_together/metrics/search_query_capture_service.rb) normalizes queries, respects platform-level analytics settings, and can hash retained search text with a `sha256:` prefix.
- [TrackSearchQueryJob](../../app/jobs/better_together/metrics/track_search_query_job.rb) and the metrics system docs tie search-query retention into the analytics/privacy lane.
- [Search](../../app/services/better_together/search.rb) and [ElasticsearchBackend](../../app/services/better_together/search/elasticsearch_backend.rb) make the backend selection explicit.
- [Registry](../../app/services/better_together/search/registry.rb) defines the searchable model surface used by the global search lane.
- The real-backend validation path is covered by [search_matching_spec.rb](../../spec/integration/better_together/search_matching_spec.rb) and the search-audit / health work called out in the compare range.
