# 0.11.0 Feature Gating Assessment

**Status:** Assessment complete  
**Date:** 2026-06-01  
**Scope:** Analysis of release/0.11.0-notes branch diff against main to identify incomplete, commented-out, and management-gated features for alpha/beta feature gating.

---

## Executive Summary

The `0.11.0` branch ships several intentionally gated or deferred features that should be added to the feature registry for controlled alpha/beta rollout:

1. **E2EE Messaging** — Signal Protocol encryption, currently disabled by environment variable
2. **Robot/AI Runtime** — Autonomous agent configuration and access controls, intentionally documentation-first
3. **12 New Content Block Types** — Implemented but page-builder rollout deferred until 0.11.x patch review
4. **Profiling Libraries** — Optional profiling tooling kept disabled in production by default
5. **Partially Completed Features** — DocumentationBuilder, search suggestions, calendar/conversation associations

---

## 1. E2EE (End-to-End Encrypted) Messaging

### Current State

**Implementation Status:** COMPLETE but GATED

- Signal Protocol E2E encryption for conversations fully implemented
- `EncryptedConversation` model with key exchange and sealed-sender delivery
- E2EE session bootstrap, key management, and backup/restore flows
- Client-side Stimulus controllers for key setup and message encryption
- Server-side key storage and per-conversation encryption state

**Current Gating Method:**
```ruby
# In BetterTogether module initialization
BetterTogether.e2ee_messaging_enabled? # Checks BETTER_TOGETHER_E2EE_MESSAGING_ENABLED env var
```

**Release Notes Position:** 0.11.0 release notes explicitly call out E2EE as "intentionally gated" and "conservative rollout."

### Recommendation

**Add to feature registry as:**
```yaml
- key: e2ee_messaging
  name: End-to-End Encrypted Conversations
  description: Signal Protocol-based E2EE for direct conversations with key management, backup, and restore flows.
  default_rollout: off
  owner_surface: both
  category: security
```

**Rationale:**
- Currently requires environment variable plus manual enablement
- Feature gating provides per-person/platform granularity instead of global toggle
- Allows alpha/beta testing with specific organizers before broad rollout
- Release notes explicitly state "limit enablement to opted-in deployments"
- Integration with existing feature access grant system for explicit person-level grants

**Migration Path:**
1. Add `e2ee_messaging` to `config/feature_gates.yml` with `default_rollout: off`
2. Create migration to mark known E2EE adopters with explicit alpha/beta grants
3. Update `BetterTogether.e2ee_messaging_enabled?` to check both env var AND feature gate
4. Deprecate env var in favor of feature gate in 0.12.0

## 2. Robot/AI Runtime Configuration

### Current State

**Implementation Status:** COMPLETE but UNDOCUMENTED IN ORGANIZER UI

- `BetterTogether::Robot` model with access token authentication
- First-party robot scopes: public, community, and private content access
- Robot authentication middleware in ApplicationController
- Separate `robots_controller.rb` for platform-scoped HTML CRUD
- `api/v1/robots_controller.rb` for JSONAPI resource endpoints
- Robot configuration resolution flows (documented in release notes)
- Provider adapter architecture for AI/translation services
- C3 contribution tracking and bot-safety baseline integration

**Current Gating Method:**
None — fully implemented but intentionally "docs-first" with no dedicated organizer admin UI in this release.

**Release Notes Position:** "AI runtime and robot configuration remain intentionally documentation-first in this packet because the shipped change is architectural and no dedicated robot admin UI ships on branch tip."

### Recommendation

**Add to feature registry as:**
```yaml
- key: robot_configuration_ui
  name: Robot Configuration Management
  description: Organizer-facing UI for creating, configuring, and managing autonomous agents and API robots.
  default_rollout: off
  owner_surface: ui
  category: automation

- key: robot_api_access
  name: Robot API Access
  description: First-party robot authentication and JSONAPI endpoints for autonomous content access.
  default_rollout: beta
  owner_surface: api
  category: automation
```

**Rationale:**
- API is already implemented and functional (beta-ready)
- Organizer UI is deferred pending security/policy review
- Separation allows API-first adoption while UI/docs stabilize
- Release notes explicitly identify this as deferred from branch-tip organizer surfaces
- Separate gating for API vs UI allows independent rollout strategy

**Notes:**
- Consider linking `robot_configuration_ui` to require platform-manager role via policy
- API feature gate already allows beta testing with trusted integrators
- Organizer UI rollout can proceed independently in 0.11.x patch cycle

## 3. Content Block Families (12 Deferred Block Types)

### Current State

**Implementation Status:** COMPLETE but INTENTIONALLY DEFERRED FROM EDITOR ROLLOUT

PR #1350 added 12 new structured content block types. All are fully implemented with models, views, fields, partials, and tests. However, their availability in the "Add Block" organizer editor is intentionally deferred.

**Block Types:**
1. **Accordion Block** — Collapsible accordion sections
2. **Alert Block** — Info/warning/success/danger alerts with dismissible option
3. **Call to Action Block** — Multi-layout CTA with primary/secondary buttons
4. **Checklist Block** — Interactive checklist with item tracking
5. **Communities Block** — Community collection display with item limit/view-more
6. **Events Block** — Event collection display with scope (upcoming/recent/all)
7. **Navigation Area Block** — Navigation menu builder
8. **People Block** — People/user directory display
9. **Posts Block** — Posts collection with scope (published/all)
10. **Quote Block** — Formatted quotation display
11. **Statistics Block** — Statistics/metrics display
12. **Video Block** — Video embed with aspect ratio and caption

**Current Status in Code:**
- All have `content_addable? = true` (enabled in 0.11.0)
- All views render correctly
- All specs pass
- Changelog mentions: "page-builder rollout deferred until a 0.11.x patch review (#1350)"

**Latest Changes:**
- Commit `b44fe6cda` ("enable all blocks + resolve PR #1492 review findings") enabled all blocks
- PR #1492 resolved findings; all blocks now enabled by default

### Assessment Conflict

**⚠️ Status Change During Release Cycle:**
The blocks were initially deferred but were re-enabled in the lead-up to PR #1492. However, the release notes still describe the deferred state. This represents a scope shift that should be clarified in release documentation.

### Recommendation

**Option A: Keep All Blocks Enabled (Current State)**

If the decision is to ship all blocks in 0.11.0, no feature gating is needed. However, release notes should be updated to reflect this.

**Option B: Re-defer Selected Newer Block Types (Recommended for Conservative Rollout)**

If organizer UI stability is a concern, consider gating the more complex blocks:

```yaml
# Deferred blocks (resource collections + newer blocks)
- key: content_blocks_advanced
  name: Advanced Content Blocks
  description: |
    Structured content blocks for collections (Communities, Events, People, Posts), 
    interactive elements (Accordion, Checklist, Statistics, CTA), 
    and embeds (Video, Quote, Alert).
  default_rollout: alpha
  owner_surface: ui
  category: content
```

Then in the block organizer UI (`app/controllers/better_together/content/blocks_controller.rb`):

```ruby
# Only show blocks if available in the editor
def available_block_types
  all_blocks = BetterTogether::Content::Block.descendants
  
  advanced_block_classes = [
    BetterTogether::Content::AccordionBlock,
    BetterTogether::Content::AlertBlock,
    # ... etc
  ]
  
  all_blocks.reject do |klass|
    advanced_block_classes.include?(klass) && 
      !feature_enabled?('content_blocks_advanced')
  end
end
```

**Rationale for Option B:**
- Release notes explicitly mention deferred block rollout
- Commit history shows blocks were intentionally deferred, then re-enabled
- Gives organizers time to stabilize with core block set (text, image, mermaid, rich-text)
- More complex blocks (Communities, Events, Posts) may benefit from additional UI review
- Conservative default allows slower adoption curve

### Current Code State

All blocks are enabled (`content_addable? = true`). To implement Option B deferral:

1. Modify `app/models/better_together/content/block.rb`:
   ```ruby
   def self.content_addable?
     true  # or override in subclass to false for deferred blocks
   end
   ```

2. Add a feature gate check in the editor view to hide unavailable blocks.

## 4. Profiling Module

### Current State

**Implementation Status:** IMPLEMENTED with PRODUCTION DISABLE

Located in `app/lib/better_together/profiling.rb`:

```ruby
module BetterTogether
  # Central toggle for optional profiling libraries that should stay disabled
  # in normal production traffic unless an operator explicitly enables them.
  module Profiling
    # Profiling utilities...
  end
end
```

**Current Gating Method:**
- Module exists but profiling is disabled by default in production
- No environment variable or configuration currently enables it

### Recommendation

**Add to feature registry as:**
```yaml
- key: profiling_enabled
  name: Production Profiling
  description: Enable performance profiling libraries (MemoryProfiler, StackProf, etc.) in production. For debugging only.
  default_rollout: off
  owner_surface: both
  category: debugging
```

**Rationale:**
- Profiling is inherently risky in production (performance impact, data exposure)
- Feature gating provides fine-grained control per-person rather than global toggle
- Allows platform organizers to enable profiling for specific debugging sessions
- Clear audit trail of who enabled profiling and when
