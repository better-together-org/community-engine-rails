# 0.11.0 AI runtime and robot configuration

This note explains the AI/runtime changes now included in the `0.11.0` release range.

Unlike the more visible federation, privacy, and content tooling work, this part of the release is mostly architectural. It matters because it changes how Community Engine chooses AI providers and models, how translation work is routed, and how future AI-capable features can be configured and audited without hard-wiring the engine to one direct provider path.

## Quick read

- CE no longer needs to think in terms of one built-in OpenAI-only runtime path.
- AI-capable bots now resolve through persisted robot configuration plus adapterized LLM/embedding dispatch.
- Translation remains the main user-facing AI feature on branch tip, but the runtime beneath it is now more flexible and more auditable.
- This is groundwork for platform-specific AI behavior and future provider extraction, not a new end-user AI management UI.

## Who should read this

- **Stakeholders and reviewers:** read the overview and “why this matters” sections.
- **Platform organizers/operators:** read the runtime impact and operational implications sections.
- **Developers/maintainers:** read the implementation surfaces and architecture notes sections.

## What changed

### 1. Persisted robot records were introduced

The release range adds a `BetterTogether::Robot` model and migration so CE can persist AI runtime configuration instead of relying only on environment-driven defaults.

That means CE can now store, for a named robot:

- the provider key
- the default model
- the default embedding model
- optional system prompt text
- settings used by downstream runtime dispatch
- whether the robot is platform-specific or global

This gives the runtime a stable configuration layer above raw provider credentials.

### 2. Robot resolution is now platform-aware

Robot resolution is designed to prefer a platform-scoped active robot first, then fall back to an active global robot with the same identifier.

In practice, this means:

- one platform can run with a different AI profile than another
- CE can keep a global fallback when no platform-specific robot exists
- AI behavior is better aligned with the broader platform-aware direction of `0.11.0`

### 3. LLM and embeddings dispatch were adapterized

The release range also adds adapterized dispatch for:

- chat/completion-style calls
- embeddings calls

Instead of encoding all runtime behavior directly in one provider-specific path, CE now routes through the shared adapter registry and default adapter implementations. This makes it easier to swap or extend provider integrations without treating one vendor path as the whole system.

### 4. `ruby_llm` replaced the older direct bootstrap path

The old direct OpenAI initialization path was removed in favor of a `ruby_llm` initializer and adapter-backed defaults.

This is important because the runtime contract is now closer to:

`TranslationBot` / other bots -> `ApplicationBot` -> robot resolution -> adapter dispatch

rather than:

bot code -> one directly initialized provider client

### 5. Translation still matters, but the release value is deeper than translation UI

The visible AI surface on branch tip is still primarily translation. The bigger release significance is the runtime beneath translation:

- provider/model selection is less hard-coded
- audit context can travel with robot/runtime decisions
- CE is better positioned for future local or alternative provider routing
- platform-specific AI behavior becomes a first-class concept instead of a future workaround

## Why this matters

### For stakeholders

This change reduces architectural lock-in. Even if the visible UI still looks mostly like “translation,” the system underneath is moving toward a safer long-term model where provider choice, model choice, and runtime defaults are not hidden inside one hard-coded integration path.

### For organizers and operators

This makes future AI behavior easier to reason about at the platform level. The important shift is not “there is a robot admin screen now” — there is not — but rather that the engine now has a data model and runtime contract for platform-aware AI configuration.

### For developers and maintainers

This is the foundation for future provider extraction and audited runtime routing. It introduces a cleaner separation between:

- runtime configuration (`BetterTogether::Robot`)
- bot behavior (`ApplicationBot`, `TranslationBot`)
- provider invocation (`BetterTogether` adapter registry and default adapters)

That separation should make future AI work easier to test, document, and evolve.

## Operational implications

The release does **not** yet ship a complete organizer-facing robot administration UI.

So for `0.11.0`, the right way to describe this is:

- **implemented runtime foundation:** yes
- **persisted robot model:** yes
- **platform-aware resolution contract:** yes
- **visible configuration UI for organizers:** not yet first-class on branch tip

This is why the `0.11.0` release notes treat the topic as documentation-first rather than screenshot-heavy.

## Relationship to the rest of 0.11.0

This AI/runtime work fits the same larger release themes already visible elsewhere in the packet:

- **platform awareness:** robot resolution can now be platform-scoped
- **auditability and governance:** runtime choices are easier to reason about and inspect
- **adapterization:** CE is moving toward subsystem contracts instead of one-off direct integrations
- **future-safe operations:** provider choice and runtime defaults are less entangled with one specific implementation
