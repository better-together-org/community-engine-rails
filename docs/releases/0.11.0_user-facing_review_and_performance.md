# 0.11.0 User-Facing Review and Performance Report

This report compares `v0.10.0..release/0.11.0-notes` at the current release-branch tip and focuses on the surfaces most likely to change what people see and how quickly those pages respond.

The review emphasizes navigation items, posts, and adjacent organizer/member flows, then cross-checks those changes against the currently available performance evidence for `communityengine.app`.

## User-facing delta worth calling out

### 1. Navigation and host dashboard review paths expanded

`0.11.0` makes the host/organizer navigation tree materially larger and more stateful than it was at `0.10.0`.

What changed:

- [NavigationItem.route_names](../../app/models/better_together/navigation_item.rb) now includes additional review and operations surfaces such as:
  - `host_dashboard_platform_connection_review`
  - `host_dashboard_safety_review`
  - `platform_connections`
  - `reports`
  - `safety_cases`
- [NavigationItemsHelper](../../app/helpers/better_together/navigation_items_helper.rb) now:
  - builds visibility-aware cache keys instead of relying on path-specific cache fragments
  - memoizes visibility and child lookups per user/platform context
  - routes the host dashboard review tabs through a dedicated turbo frame
  - adds special permission-aware visibility rules for federation review and safety review items

Why that matters:

- The branch is clearly moving the host dashboard toward a review console for federation and safety, not just a generic admin index.
- The navigation system is now carrying more authorization logic, more tree depth, and more cache-context variation in the request path itself.

### 2. Posts moved toward governed contributions rather than simple inline authorship

The post surface is not just a styling refresh. It shifts how authorship and supporting evidence are presented and edited.

What changed:

- [PostsController](../../app/controllers/better_together/posts_controller.rb) now wraps `create` and `update` in `Authorship.with_creator(...)` and preloads post cards with `resource_class.card_render_includes`.
- [Post.card_render_includes](../../app/models/better_together/post.rb) now preloads:
  - translations
  - cover-image blobs
  - contributions and authors
  - category cover images
  - rich text content when present
- [posts/_form](../../app/views/better_together/posts/_form.html.erb) replaces the old multi-select author picker with `governed_contributions_fieldset(...)`.
- The translated rich-text field now receives `citation_options`, which aligns post editing with the broader governed-contributions / citation lane.
- [posts/_post](../../app/views/better_together/posts/_post.html.erb) and [posts/show](../../app/views/better_together/posts/show.html.erb) remove the old inline author chips from the card/show presentation, while the show toolbar now renders shared content actions in the header.

Why that matters:

- The visible post UX now presents contribution governance more deliberately and reduces lightweight inline authorship decoration.
- This aligns the post editor and post display with the branch-wide move away from public evidence/citation surfaces that were not yet ready for broad rollout.

### 3. Adjacent surfaces widened the amount of visible review-state UI

Even when the user specifically lands on navigation or posts, the surrounding release delta increases the amount of review-state UI that users and organizers encounter.

Examples from [routes](../../config/routes.rb):

- membership-request review paths under communities and platforms
- platform connection review and approval paths
- report index/show/followup paths
- person data export and deletion-request paths
- settings/my-data path
- search health endpoint in the metrics area

Why that matters:

- These additions make the branch more review-oriented and more operationally explicit.
- They also increase the number of conditional nav entries and page states that have to render quickly under permission-aware layouts.

## Performance findings

## Target

The current expectation is that user-facing pages should stay at or below roughly **350 ms**, and that even the slowest normal pages should not drift into the **750 ms** range.

## MiniProfiler findings

The MiniProfiler capture shows several routes well above both thresholds.

| Surface | Observed duration | Notes |
| --- | ---: | --- |
| `/en/host/metrics/reports` | 6176.7 ms | Main HTML request |
| `/en/host/metrics/link_checker_reports` | 2890.5 ms | Report lane still expensive |
| `/en/host/metrics/user_account_reports` | 1715.6 ms | Above acceptable ceiling |
| `/en/host/metrics/page_view_reports` | 1464.2 ms | Above acceptable ceiling |
| `/en/host/metrics/link_click_reports` | 1360.1 ms | Above acceptable ceiling |
| `/en` | 1220.8 ms and 1379.2 ms | Homepage still above target |
| `people#show` request | 41045.5 ms | Severe outlier in the same profiler bundle |

The two most relevant request-path details from the capture are:

- `bettertogether::metrics::reports#index` shows **74 SQL** and about **1394.0 ms** of query time during the worst `/en/host/metrics/reports` request.
- `people/show.html.erb` shows **89 SQL** and about **564.8 ms** query time inside the rendered profile view, despite recent preload work on the branch.

The layout/navigation cost is also still high inside the metrics dashboard request:

| Partial | Total time | SQL count | SQL time |
| --- | ---: | ---: | ---: |
| `better_together/_host_nav.html.erb` | 1205.9 ms (573.6 self) | 8 | 252.7 ms |
| `better_together/_host_sidebar_nav.html.erb` | 441.4 ms | 12 | 208.5 ms |

That means the host-nav lane alone accounts for roughly **20 SQL queries** and about **461.2 ms** of query time during the slow metrics page.

## Sentry findings

Sentry access worked, but the performance query returned no transaction rows for `community-engine-rails`:

- with `--environment production`
- and again without an environment filter

That absence is itself a finding:

- either transaction performance is not being captured for the relevant runtime
- or the environment/project mapping for `communityengine.app` is incomplete
- or performance sampling is too sparse to support release review

So the profiler artifact is currently the stronger evidence source for this review, and Sentry should be treated as an instrumentation gap rather than a contradictory data point.

## Code correlation

### Metrics dashboard cost is still front-loaded into the HTML request

[ReportsController](../../app/controllers/better_together/metrics/reports_controller.rb) builds a large amount of data in `set_metrics_data` before the page renders:

- page views
- link clicks
- downloads
- shares
- link-checker summaries
- search-query summaries
- user-account growth summaries
- search-health audit data

This branch already includes [feat(metrics): preload report charts and refine search health](../../app/controllers/better_together/metrics/reports_controller.rb), but the profile shows that the HTML request still pays the up-front aggregation cost rather than treating more of the dashboard as lazy or deferred.

### Navigation is better cached than before, but it is still too expensive on review-heavy pages

The nav helper clearly improved between `0.10.0` and the release branch:

- fragment cache keys now include locale/platform/community/visibility context
- visibility checks are memoized
- host dashboard review items are explicitly modeled

Even so, the profiler shows the host nav and host sidebar still consuming enough query time to violate the 350 ms ceiling by themselves on the slow metrics route.

That suggests the release branch fixed part of the old nav cost, but the combination of:

- recursive tree visibility checks
- permission-aware child filtering
- larger review-oriented nav trees
- host dashboard sidebar rendering

is still too expensive under real request conditions.

### Person profile work improved one hotspot but did not eliminate it

This branch includes:

- [perf: reduce N+1 queries on platform lookup and person profile page](../../app/controllers/better_together/people_controller.rb)
- [fix(performance): preload person and post surfaces](../../app/controllers/better_together/people_controller.rb)

[PeopleController](../../app/controllers/better_together/people_controller.rb) now preloads memberships, roles, joinables, contact details, contributed pages, and contributed posts, but `people/show` still shows severe render and query cost in the profiler bundle.

The remaining cost is likely in one or more of:

- tabbed profile content with many conditional collections
- toolbar / content-action rendering
- calendar/event collection rendering
- contributed cards and other nested partials that still do per-item work outside the current preload set

### Posts are likely a secondary contributor, not the main release-line bottleneck

The post surface itself shows the branch moving in the right direction:

- post cards now preload more data
- authorship is explicit and creator-backed
- the editor uses governed contributions

That means posts matter to the user-facing review, but they are less likely than metrics, nav, and people profile rendering to be the main reason many pages are landing near or above 750 ms.

## Remediation priorities

### 1. Split the metrics dashboard into a lighter initial HTML payload

Highest priority.

The current metrics dashboard appears to do too much work before first paint. Likely next steps:

- render the shell quickly and lazy-load more chart datasets
- defer search-health audit work behind explicit user action or separate caching
- cache date-bounded aggregate datasets rather than recomputing everything on each dashboard request

### 2. Re-profile and trim the host nav path

Highest priority for shared layout performance.

Recommended follow-up:

- instrument `visible_nav_items_for` and child filtering counts
- confirm whether nav queries still miss a preload/caching opportunity for host review trees
- consider precomputing the host dashboard sidebar tree per visibility context instead of recursively filtering during render

### 3. Run a dedicated second-pass profile on `people#show`

The person profile remains a major outlier even after recent preload fixes.

Recommended follow-up:

- isolate which tab contents still query during render
- inspect contributed posts/pages and calendar sections under a focused profile
- verify whether toolbar/content-action helpers trigger additional lookups not covered by current controller preloads

### 4. Restore usable Sentry transaction coverage for `communityengine.app`

Without Sentry transaction data, release review and regression watching are too dependent on one-off MiniProfiler captures.

Recommended follow-up:

- confirm the correct Sentry project/environment for the deployed app
- verify performance sampling is enabled for web transactions
- ensure host-dashboard and profile routes emit transaction events consistently

## Reviewer note

The branch already includes real performance work for nav rendering, metrics reporting, people profiles, and posts. The problem is not lack of effort on the release line. The problem is that the remaining hot paths are still far above the target envelope in the current evidence.

In practice, this means the release review should treat performance as an open release-quality concern for `communityengine.app`, especially around:

- host dashboard metrics routes
- shared host navigation rendering
- person profile pages
