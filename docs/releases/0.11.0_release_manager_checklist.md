# 0.11.0 Release-Manager Checklist

Use this checklist when treating the `release/0.11.0-notes` branch as a release candidate. The branch is not just a documentation packet: it also carries runtime changes, migrations, CI/runtime default changes, and intentionally gated surfaces that need an explicit operator story.

## 1. Confirm the release story matches the branch

- `CHANGELOG.md` and `docs/releases/0.11.0.md` describe the same shipped scope.
- Deferred or gated surfaces are labeled as such instead of being described like normal organizer-ready features.
- Search docs describe the branch default honestly: `pg_search` is the shipped default, not Elasticsearch.

## 3. Confirm changed defaults and runtime assumptions

- `.env.dev` now defaults `SEARCH_BACKEND='pg_search'`.
- The Rails app search service defaults to `BetterTogether::Search::PgSearchBackend`.
- Old Elasticsearch wording has been removed or clearly reframed as optional/historical where it is no longer the primary branch path.
- CI/runtime notes do not imply an Elasticsearch service requirement when the branch test path now uses the pg_search lane.

## 4. Verify gated and deferred rollout posture

These surfaces are intentionally not "ship it everywhere by default" features:

- **E2EE conversations:** documentation-first and rollout-gated
- **AI/robot runtime foundations:** implemented as architecture/runtime groundwork; no first-class organizer UI on branch tip
- **new structured content block families:** implemented in code but deferred from the normal `Add Block` organizer rollout pending a later `0.11.x` review

Release docs and PR messaging should keep these boundaries explicit.

## 5. Post-upgrade verification checklist

After migrations and asset/docs refresh:

- run the release branch test/lint/security lanes that are meant to support `0.11.0`
- verify docs validation and diagram rendering
- verify referenced screenshots/diagram exports still exist on branch tip
- verify search docs, changelog wording, and runtime defaults all agree on `pg_search`
- verify the release packet still labels docs-first sections honestly

## 6. Reviewer handoff

If this branch remains one large PR, the PR body should give reviewers a simple scope map:

1. release packet and evidence assets
2. runtime and upgrade surface
3. shipped product/runtime slices
4. deferred or gated slices

That map is the minimum needed to make a mixed release branch reviewable without splitting it first.
