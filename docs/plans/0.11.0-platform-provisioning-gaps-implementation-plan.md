## PR B тАФ Request spec for the provisioning + domain flow

**Risk: low (test-only, no production code). File: 1 new spec.**

### File

`spec/requests/better_together/platform_provisioning_with_domain_spec.rb`

(Note: the original punch list suggested `spec/system/...`. That's corrected here тАФ see rationale below.)

### Why a request spec, not a system spec

Confirmed by reading `spec/rails_helper.rb`, `spec/support/capybara.rb`, and all 4 existing files under `spec/system/better_together/`: this codebase has **no** `Capybara.app_host`, `/etc/hosts` manipulation, or any other technique for making a real Selenium-driven browser resolve a fake hostname like `demo-tenant.btsdev.ca` to the local test server тАФ `Capybara.server_host` is hardcoded to `127.0.0.1`. The four existing system specs are all real-browser (`:js`), same-hostname UI-interaction tests (CSS block CRUD, invitation resend) тАФ none simulate multi-host routing, and none could.

The actual, established technique for exercising `PlatformDomain`/`PlatformContextMiddleware` end-to-end is `host!` inside a `type: :request` spec тАФ used identically in three existing sibling specs:

```ruby
# spec/requests/better_together/multi_platform_isolation_spec.rb:25 (existing, for reference)
host! 'host.example.test'
get better_together.home_page_path(locale: I18n.default_locale)
```

`host!` is `ActionDispatch::Integration::Session#host!`, available in request specs via `rspec-rails`. There is also no PlatformDomain admin UI yet (that's PR C, below) тАФ so there is nothing to click through with Capybara even if hostname simulation worked. A request spec calling the service/model layer directly, exactly matching the already-*validated* runbook in `docs/production/multi_platform_deployment.md`, is both the correct and the only feasible design.

### Spec design

```ruby
# frozen_string_literal: true

require 'rails_helper'

RSpec.describe 'Platform provisioning with domains', :skip_host_setup do
  # Mirrors the manually-validated flow in docs/production/multi_platform_deployment.md
  # ("Validated Provisioning Walkthrough (tested 2026-07-17)"), converted to a
  # permanent spec so the flow is protected by CI going forward.

  it 'provisions a tenant platform on a subdomain, attaches a custom domain, ' \
     'and keeps both hostnames isolated from an unrelated tenant' do
    host_result = BetterTogether::TenantPlatformProvisioningService.call(
      name: 'Host', host_url: 'https://host.example.test', time_zone: 'UTC', host: true
    )
    expect(host_result).to be_success

    tenant_result = BetterTogether::TenantPlatformProvisioningService.call(
      name: 'Tenant A', host_url: 'https://tenant-a.example.test', time_zone: 'UTC'
    )
    expect(tenant_result).to be_success
    tenant_platform = tenant_result.platform

    custom_domain = BetterTogether::PlatformDomain.create!(
      platform: tenant_platform, hostname: 'app.customclientdomain.test',
      active: true, share_domain: false
    )
    expect(custom_domain.primary_flag?).to be false # a primary already exists from the auto-callback

    other_result = BetterTogether::TenantPlatformProvisioningService.call(
      name: 'Tenant B', host_url: 'https://tenant-b.example.test', time_zone: 'UTC'
    )
    other_platform = other_result.platform

    # Resolution: both hostnames on Tenant A route to the same platform
    expect(BetterTogether::PlatformDomain.resolve('tenant-a.example.test').platform_id).to eq(tenant_platform.id)
    expect(BetterTogether::PlatformDomain.resolve('app.customclientdomain.test').platform_id).to eq(tenant_platform.id)
    expect(BetterTogether::PlatformDomain.resolve('tenant-b.example.test').platform_id).to eq(other_platform.id)

    # Isolation: same slug on two platforms, each hostname only ever serves its own platform's page
    page_a = create(:better_together_page, platform: tenant_platform, slug: 'shared-slug',
                                            privacy: 'public', published_at: 1.day.ago)
    page_b = create(:better_together_page, platform: other_platform, slug: 'shared-slug',
                                            privacy: 'public', published_at: 1.day.ago)

    host! 'tenant-a.example.test'
    get better_together.render_page_path(locale: I18n.default_locale, id: 'shared-slug')
    expect(response.body).to include(page_a.title)
    expect(response.body).not_to include(page_b.title)

    host! 'app.customclientdomain.test'
    get better_together.render_page_path(locale: I18n.default_locale, id: 'shared-slug')
    expect(response.body).to include(page_a.title) # custom domain resolves to the SAME tenant platform

    host! 'tenant-b.example.test'
    get better_together.render_page_path(locale: I18n.default_locale, id: 'shared-slug')
    expect(response.body).to include(page_b.title)
    expect(response.body).not_to include(page_a.title)
  end
end
```

Notes on this draft (to be finalized by whoever implements PR B, not final code):
- `:skip_host_setup` tag mirrors `multi_platform_isolation_spec.rb`'s own header тАФ confirm the exact tag name and its effect (`spec/support/automatic_test_configuration.rb`) still matches at implementation time, since this spec deliberately provisions its own host platform rather than using the auto-configured one.
- `render_page_path` route helper name should be double-checked against `config/routes.rb` at implementation time (used here based on `multi_platform_isolation_spec.rb`'s own usage).
- `create(:better_together_page, ...)` тАФ do not pass `platform:` via the `:with_community` trait; pass `platform:` directly, matching the factory's documented fallback-avoidance pattern (`spec/factories/better_together/pages.rb`).
- Do **not** use the `:primary` trait on the `platform_domains` factory for the custom-domain step тАФ a primary domain already exists from the tenant platform's `sync_primary_platform_domain!` callback; passing `:primary` again raises `ActiveRecord::RecordInvalid` per the `only_one_primary_flag` validation (confirmed in `spec/requests/better_together/platform_domain_resolution_spec.rb`).

### PR #1581 signature dependency

**Check PR #1581's merge status before implementing.** If merged, `TenantPlatformProvisioningService.call` uses `steward:`/`privacy:` (not `admin:`), and the calls above don't need `admin:` anyway since this spec doesn't exercise the steward-creation path тАФ but do re-verify the `time_zone`/`privacy` defaults didn't change in a way that affects the `host_url`-uniqueness assumptions above. If still open, no change needed to the draft above.

No `:js` tag, runs at normal request-spec speed. No CI accommodation needed.

## PR C тАФ `PlatformDomainsController` + policy + views + i18n + route

**Risk: medium (new feature surface, no data model changes). Files touched: ~12 new, 2 edited.**

### Why this pattern, not `PlatformConnectionsController`

`PlatformConnection` is a **flat, cross-platform** resource (it links a `source_platform` to a `target_platform`, so it can't be nested under a single `platform_id` route segment) with a **global** Pundit policy (`PlatformConnectionPolicy` checks `permitted_to?('manage_network_connections')` with no `record` argument, i.e. not scoped to any specific platform).

`PlatformDomain` is the opposite shape: `belongs_to :platform` with a real `has_many :platform_domains, dependent: :destroy` on `Platform` (confirmed, `app/models/better_together/platform.rb:116`). That is structurally identical to `StorageConfiguration` and `FeatureAccessGrant`, both already nested under `resources :platforms do ... end` in the `host`-scoped, `manage_platform`-constrained section of `config/routes.rb` (confirmed, lines ~477-494):

```ruby
# config/routes.rb (confirmed existing, for reference)
resources :platforms, only: %i[index show new create edit update] do
  member do
    get :available_people
  end
  resources :feature_access_grants, except: :show
  resources :robots, only: %i[index new create edit update destroy]
  resources :person_platform_memberships
  resources :platform_invitations, only: %i[create destroy] do
    member { put :resend }
  end
  resources :storage_configurations, except: %i[show] do
    member { put :activate }
  end
end
```

So the design below follows `StorageConfigurationsController`/`FeatureAccessGrantPolicy`, not `PlatformConnectionsController`/`PlatformConnectionPolicy`.

### C1. Route

Add to the block above, alongside `storage_configurations`:

```ruby
resources :platform_domains, except: %i[show]
```

No member actions needed (see C2 тАФ `PlatformDomain` already self-enforces primary/share exclusivity on save, unlike `StorageConfiguration`'s separate `activate` action). Produces `platform_platform_domains_path(@platform)`, `new_platform_platform_domain_path(@platform)`, `edit_platform_platform_domain_path(@platform, domain)`, `platform_platform_domain_path(@platform, domain)`.

### C2. Controller тАФ `app/controllers/better_together/platform_domains_controller.rb`

Modeled directly on `StorageConfigurationsController` (full file confirmed above), stripped of the storage-specific `activate`/`rebind_active_storage_service`/`touch_restart_txt` machinery (not applicable here):

```ruby
# frozen_string_literal: true

module BetterTogether
  # Admin CRUD for platform domains (subdomain aliases and custom domains).
  # Nested under platforms тАФ all actions are scoped to a specific platform.
  # Requires manage_platform permission (enforced via route constraint + Pundit).
  class PlatformDomainsController < ApplicationController
    before_action :set_platform
    before_action :set_platform_domain, only: %i[edit update destroy]
    after_action :verify_authorized

    def index
      authorize PlatformDomain
      @platform_domains = @platform.platform_domains.order(primary_flag: :desc, created_at: :asc)
    end

    def new
      authorize @platform.platform_domains.build
    end

    def create
      @platform_domain = @platform.platform_domains.build(platform_domain_params)
      authorize @platform_domain

      if @platform_domain.save
        redirect_to platform_platform_domains_path(@platform),
                    status: :see_other,
                    notice: t('better_together.platform_domains.created')
      else
        render :new, status: :unprocessable_entity
      end
    end

    def edit
      authorize @platform_domain
    end

    def update
      authorize @platform_domain

      if @platform_domain.update(platform_domain_params)
        redirect_to platform_platform_domains_path(@platform),
                    status: :see_other,
                    notice: t('better_together.platform_domains.updated')
      else
        render :edit, status: :unprocessable_entity
      end
    end

    def destroy
      authorize @platform_domain

      if @platform_domain.primary_flag?
        redirect_to platform_platform_domains_path(@platform),
                    status: :see_other,
                    alert: t('better_together.platform_domains.cannot_destroy_primary')
        return
      end

      @platform_domain.destroy
      redirect_to platform_platform_domains_path(@platform),
                  status: :see_other,
                  notice: t('better_together.platform_domains.destroyed')
    end

    private

    def set_platform
      @platform = Platform.friendly.find(params[:platform_id])
    end

    def set_platform_domain
      @platform_domain = @platform.platform_domains.find(params[:id])
    end

    def platform_domain_params
      params.require(:platform_domain).permit(:hostname, :primary_flag, :share_domain, :active)
    end
  end
end
```

Notes:
- `authorize @platform.platform_domains.build` in `new` (rather than `authorize PlatformDomain`, the class-level check `StorageConfigurationsController` uses) тАФ needed because `PlatformDomainPolicy` (C3 below) is **record-scoped** (`record.platform`), unlike `StorageConfigurationPolicy` which is a simple global `platform_manager?` check. This matches `FeatureAccessGrantsController`'s own convention of building a scratch record before authorizing, per the research findings.
- `destroy` guards against deleting the primary domain (mirroring `StorageConfigurationsController#destroy`'s guard against destroying the active storage config) тАФ the model's `PrimaryFlag` concern doesn't itself prevent this, so the controller should. Confirm this guard is actually desired at implementation time (it's a UX safety net, not a hard model invariant) тАФ alternative: let the destroy proceed and require the steward to explicitly set a new primary first, matching stricter interpretations of "primary must always exist." Flagging as a design decision for the implementer, not settling it here.
- No Turbo Stream handling тАФ matches `storage_configurations`/`feature_access_grants` convention (plain `format.html`), not the generic `ResourceController` base's turbo_stream branch.

### C3. Policy тАФ `app/policies/better_together/platform_domain_policy.rb`

Modeled directly on `FeatureAccessGrantPolicy` (full file confirmed above):

```ruby
# frozen_string_literal: true

module BetterTogether
  # Policy for host-admin managed platform domains.
  # Follows the existing /host management contract, scoped per-platform so a
  # steward of one platform cannot manage another platform's domains.
  class PlatformDomainPolicy < ApplicationPolicy
    def index?;   platform_manager_for_target_platform?; end
    def new?;     platform_manager_for_target_platform?; end
    def create?;  platform_manager_for_target_platform?; end
    def edit?;    platform_manager_for_target_platform?; end
    def update?;  platform_manager_for_target_platform?; end
    def destroy?; platform_manager_for_target_platform?; end

    private

    def target_platform
      record.respond_to?(:platform) ? record.platform : nil
    end

    def platform_manager_for_target_platform?
      return false unless agent.present? && target_platform.present?

      agent.permitted_to?('manage_platform', target_platform) ||
        agent.permitted_to?('manage_platform_settings', target_platform)
    end
  end
end
```

For the `index?` check in the controller's `index` action (`authorize PlatformDomain` тАФ a class, not an instance, since there's no natural single record to scope `index?` to when listing), this policy's `target_platform` would resolve to `nil` (a class doesn't `respond_to?(:platform)`), which fails closed (`return false`). **This needs a decision at implementation time**: either (a) change the controller's `index` action to `authorize @platform.platform_domains.build` (matching `new`, giving `index?` a real `record.platform` to check тАФ the resulting built-but-unsaved record's `platform` is set correctly since it comes from `@platform.platform_domains.build`), or (b) give `PlatformDomainPolicy` a `Scope`/index-specific override that takes `@platform` from context another way. Recommend (a) тАФ it's the smaller change and matches the `new?` pattern already in this same policy.
