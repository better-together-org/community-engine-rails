### P0-2. Stale-container `FederatedContentPullJob` incident â€” confirmed resolved on this branch, watch the deploy step

The 2026-06-18 incident (documented in management-tool's `skills/ce-deployment/SKILL.md`) was `PG::UndefinedColumn: column better_together_platform_domains.primary`. Investigation for this plan confirmed:
- Migration `db/migrate/20260514100000_refactor_platform_domain_primary_to_primary_flag.rb` (present on `release/0.11.0-notes`, idempotent via `column_exists?` guards) renamed the column from `primary` to `primary_flag`, with `PlatformDomain` providing a Ruby-level `alias_attribute :primary, :primary_flag` for backward-compatible code.
- The incident's root cause was **not** a missing migration â€” it was a stale `upcoming-NNNNN` Docker container running pre-migration code against an already-migrated database (a deploy-hygiene issue, per the ce-deployment skill's own diagnosis).

*Effort:* none required here â€” the migration is correct and present. *Action:* no code change; ensure the "verify no stale `upcoming` containers" post-deploy step in `skills/ce-deployment/SKILL.md` is followed on every future deploy that includes federation code, since this failure mode will recur for *any* schema-changing migration if that step is skipped, not just this one.

## P1 â€” Release-readiness / usability gaps

### P1-1. No in-app UI for `PlatformDomain` management

There is no `PlatformDomainsController`/views to create, list, or deactivate additional domains (subdomain aliases or custom domains) on an existing platform â€” only the Rails console/rake path (see assessment Â§3a) and, once PR #1581 merges, a billing-gated *platform creation* form that still doesn't expose domain management as a distinct concern.

**Recommendation:** a small `PlatformDomainsController` (index/new/create/update/destroy, nested under an existing platform-management surface, steward/manager-authorized) exposing `hostname`, `primary_flag`, `share_domain`, `active`. Reuse `PlatformDomain`'s existing validations (uniqueness, `primary_domain_must_be_active`, `share_domain_must_be_active`) â€” no model changes needed, this is UI-only.

*Effort estimate:* smallâ€“medium (1 controller, ~4 views, policy, request spec, i18n strings â€” no new backend logic).

### P1-2. No subdomain-vs-custom-domain picker anywhere a platform/domain is created

Both the rake-task/console path and PR #1581's self-serve form take a raw `host_url`/hostname string with no guidance or validation distinguishing "give this a subdomain of our domain" from "point your own domain at it." This is a UX gap, not a data-model gap (see assessment Â§1 â€” the distinction is operational, not code-level).

**Recommendation:** once P1-1 exists, add a simple radio/toggle in the form: "Use a subdomain of `<host platform's domain>`" (auto-suffixes and validates against the host's apex) vs. "Use my own domain" (free-text hostname + a short DNS/TLS instructions panel, since a custom domain needs a client-side CNAME and its own cert per the runbook). Depends on P1-1 existing first.

*Effort estimate:* small, once P1-1 lands (mostly view work + one validation).

### P1-3. No request spec for the combined provisioning + domain flow

Confirmed no such spec exists prior to this assessment; the flow was validated manually for this assessment (see runbook) but that validation is not committed as a repeatable spec.

**Recommendation (corrected 2026-07-17):** add `spec/requests/better_together/platform_provisioning_with_domain_spec.rb` â€” **not** a `spec/system/...` browser spec as originally suggested here. Follow-up research confirmed this codebase has no `Capybara.app_host`/DNS-stubbing anywhere, so a real Selenium-driven system spec cannot resolve fake multi-tenant hostnames; the established technique for exercising `PlatformDomain`/`PlatformContextMiddleware` is `host!` inside a `type: :request` spec (see `spec/requests/better_together/multi_platform_isolation_spec.rb`). Full file-level design, including exact spec code, is in [`0.11.0-platform-provisioning-gaps-implementation-plan.md`](./0.11.0-platform-provisioning-gaps-implementation-plan.md) (PR B).

*Effort estimate:* small (the manual validation script from this assessment is a near-complete starting point â€” see the runbook's "Validated Provisioning Walkthrough" for the exact steps to convert into a spec).

### P1-4. Inconsistently platform-scoped models â€” status update: C3 is already resolved

Three groups of models are plain `ApplicationRecord` rather than `PlatformScoped`/`PlatformRecord`, despite carrying platform-shaped columns:
- `C3::Balance`, `C3::Token`, `C3::BalanceLock` (`platform_id`/`origin_platform_id` present)
- `PersonLinkedSeed` (`belongs_to :source_platform` directly)
- `Billing::Plan`, `Billing::Subscription`, `Billing::Event` (added by PR #1581; polymorphic `belongs_to :owner`)

**Update (2026-07-17):** the C3 question is **already resolved and merged**. A same-day PR (#1672, "tenant-scoping cleanup") landed on `release/0.11.0-notes` with explicit code comments on `C3::Balance`/`C3::Token` stating they are "deliberately not platform/community-scoped... a network-wide ledger by design" â€” corroborated by `docs/c3-federation-design.md` and a live `network_balance` API endpoint that intentionally sums a person's balance across platforms. `C3::BalanceLock` didn't get the matching comment in that sweep (a one-line parity gap, not an open question). `PersonLinkedSeed` and PR #1581's billing models both show strong evidence of *intentional* cross-platform design too (structural precedent from `PersonLink`/`PersonAccessGrant` for the former; explicit statements in PR #1581's own design docs plus platform-filter-free catalog queries required for its "buy a plan â†’ provision a platform" flow for the latter) â€” neither has a recorded decision comment yet, unlike C3.

**Recommendation:** no ADR or "someone decide" step needed anymore â€” the intent is clear enough in all three cases from direct evidence. What remains is documentation-only: add the same style of explanatory code comment to `C3::BalanceLock` (parity), `PersonLinkedSeed` (new), and â€” via a PR #1581 review comment rather than a direct edit, since that PR is still open â€” `Billing::Plan`/`Subscription`/`Event`. Full exact wording and file locations are in [`0.11.0-platform-provisioning-gaps-implementation-plan.md`](./0.11.0-platform-provisioning-gaps-implementation-plan.md) (PR A).

*Effort estimate:* small (documentation-only, comments only â€” no functional change, no migration).

### P1-5. No Postgres Row-Level Security

Isolation is entirely convention-based (`for_platform` scope) â€” there is no database-enforced backstop if a query forgets to scope. Given the scale of the 0.11.0 backfill (~80 models now consistently scoped), this is a reasonable accepted risk for the 0.11.0 release, but worth a forward-looking note.

**Recommendation:** not a 0.11.0 blocker. Track as a future defense-in-depth item â€” an ADR evaluating RLS policies keyed on `platform_id` for the highest-risk tables (content, messaging) would be a reasonable v0.12+ investigation.

*Effort estimate:* not scoped here â€” investigation-sized, not implementation-sized, if picked up.

### P1-6. Track PR #1581 as a provisioning-service dependency

Once PR #1581 merges, the `admin:`â†’`steward:` rename and new `privacy:` param change `TenantPlatformProvisioningService`'s public signature. Confirm at merge time that:
- `lib/tasks/better_together/provision_tenant.rake` was updated consistently (it was, per the PR's own diff, reviewed as part of this assessment).
- No other caller of the service outside the PR's own changes references the old `admin:` keyword or assumes `'UTC'`/`'public'` defaults.
- `docs/production/multi_platform_deployment.md` (updated by this assessment) gets its "pending breaking change" callout removed/updated once merged.

*Effort estimate:* small â€” a single grep-and-verify pass at merge time, plus a one-line doc edit.
