**Companion to:** [`docs/assessments/release_0_11_0_multi_platform_federation_readiness_assessment.md`](../assessments/release_0_11_0_multi_platform_federation_readiness_assessment.md) and [`docs/production/multi_platform_deployment.md`](../production/multi_platform_deployment.md)

This plan prioritizes the gaps identified in the companion assessment. Nothing here duplicates work already tracked elsewhere — where a gap is already the subject of an existing plan or issue, this document cross-references it instead of re-describing it.

## P0 — Blocks federation production activation

### P0-1. Person-level federation consent gate

**Already tracked — do not duplicate.** `docs/plans/federation-consent-identity.md` (tracking issue #1407, target v0.11.1) covers this in full: a unified `federate_content` consent setting, federated Person stubs, and a `PersonLink` claim flow. As of this writing that plan's own status line reads "Consent/identity design complete; transport T1 implementation in progress." **No `PlatformConnection` should be activated in production before this lands** — this is CE's own documented position (`docs/developers/systems/federation_system.md:118-141`), not a new finding.

*Effort:* already scoped in the referenced plan; not re-estimated here.
*Action for this release:* ensure release notes and any operator-facing federation-activation docs explicitly state this precondition, and that federation defaults to off/inactive `PlatformConnection` status until #1407 ships.

### P0-2. Stale-container `FederatedContentPullJob` incident — confirmed resolved on this branch, watch the deploy step

The 2026-06-18 incident (documented in management-tool's `skills/ce-deployment/SKILL.md`) was `PG::UndefinedColumn: column better_together_platform_domains.primary`. Investigation for this plan confirmed:
- Migration `db/migrate/20260514100000_refactor_platform_domain_primary_to_primary_flag.rb` (present on `release/0.11.0-notes`, idempotent via `column_exists?` guards) renamed the column from `primary` to `primary_flag`, with `PlatformDomain` providing a Ruby-level `alias_attribute :primary, :primary_flag` for backward-compatible code.
- The incident's root cause was **not** a missing migration — it was a stale `upcoming-NNNNN` Docker container running pre-migration code against an already-migrated database (a deploy-hygiene issue, per the ce-deployment skill's own diagnosis).

*Effort:* none required here — the migration is correct and present. *Action:* no code change; ensure the "verify no stale `upcoming` containers" post-deploy step in `skills/ce-deployment/SKILL.md` is followed on every future deploy that includes federation code, since this failure mode will recur for *any* schema-changing migration if that step is skipped, not just this one.

## P1 — Release-readiness / usability gaps

### P1-1. No in-app UI for `PlatformDomain` management

There is no `PlatformDomainsController`/views to create, list, or deactivate additional domains (subdomain aliases or custom domains) on an existing platform — only the Rails console/rake path (see assessment §3a) and, once PR #1581 merges, a billing-gated *platform creation* form that still doesn't expose domain management as a distinct concern.

**Recommendation:** a small `PlatformDomainsController` (index/new/create/update/destroy, nested under an existing platform-management surface, steward/manager-authorized) exposing `hostname`, `primary_flag`, `share_domain`, `active`. Reuse `PlatformDomain`'s existing validations (uniqueness, `primary_domain_must_be_active`, `share_domain_must_be_active`) — no model changes needed, this is UI-only.

*Effort estimate:* small–medium (1 controller, ~4 views, policy, request spec, i18n strings — no new backend logic).

### P1-2. No subdomain-vs-custom-domain picker anywhere a platform/domain is created

Both the rake-task/console path and PR #1581's self-serve form take a raw `host_url`/hostname string with no guidance or validation distinguishing "give this a subdomain of our domain" from "point your own domain at it." This is a UX gap, not a data-model gap (see assessment §1 — the distinction is operational, not code-level).

**Recommendation:** once P1-1 exists, add a simple radio/toggle in the form: "Use a subdomain of `<host platform's domain>`" (auto-suffixes and validates against the host's apex) vs. "Use my own domain" (free-text hostname + a short DNS/TLS instructions panel, since a custom domain needs a client-side CNAME and its own cert per the runbook). Depends on P1-1 existing first.

*Effort estimate:* small, once P1-1 lands (mostly view work + one validation).

### P1-3. No system/browser spec for the combined provisioning + domain flow

Confirmed no such spec exists prior to this assessment; the flow was validated manually for this assessment (see runbook) but that validation is not committed as a repeatable spec.

**Recommendation:** add `spec/system/better_together/platform_provisioning_with_domain_spec.rb` mirroring the six steps executed in the validated runbook (provision host → provision tenant on a subdomain → attach a custom domain → resolve both hostnames → verify `Page` isolation between two tenants). This is the highest-value single test to add — it directly covers the exact flow this assessment was commissioned to verify.

*Effort estimate:* small (the manual validation script from this assessment is a near-complete starting point — see the runbook's "Validated Provisioning Walkthrough" for the exact steps to convert into a spec).

### P1-4. Inconsistently platform-scoped models — needs an explicit decision, not necessarily a fix

Three groups of models are plain `ApplicationRecord` rather than `PlatformScoped`/`PlatformRecord`, despite carrying platform-shaped columns:
- `C3::Balance`, `C3::Token`, `C3::BalanceLock` (`platform_id`/`origin_platform_id` present)
- `PersonLinkedSeed` (`belongs_to :source_platform` directly)
- `Billing::Plan`, `Billing::Subscription`, `Billing::Event` (added by PR #1581; polymorphic `belongs_to :owner`)

**Recommendation:** for each group, get an explicit yes/no decision recorded (likely in a short ADR or a comment block on the model) on whether cross-platform scope is intentional. C3 and billing plans plausibly *should* be host-platform-level/network-wide by design; if so, document that decision so it isn't re-flagged as a bug in a future audit. If any turn out to be unintentional gaps, scope a follow-up migration.

*Effort estimate:* small (documentation-only) unless a genuine gap is found, in which case scope separately per model.

### P1-5. No Postgres Row-Level Security

Isolation is entirely convention-based (`for_platform` scope) — there is no database-enforced backstop if a query forgets to scope. Given the scale of the 0.11.0 backfill (~80 models now consistently scoped), this is a reasonable accepted risk for the 0.11.0 release, but worth a forward-looking note.

**Recommendation:** not a 0.11.0 blocker. Track as a future defense-in-depth item — an ADR evaluating RLS policies keyed on `platform_id` for the highest-risk tables (content, messaging) would be a reasonable v0.12+ investigation.

*Effort estimate:* not scoped here — investigation-sized, not implementation-sized, if picked up.

### P1-6. Track PR #1581 as a provisioning-service dependency

Once PR #1581 merges, the `admin:`→`steward:` rename and new `privacy:` param change `TenantPlatformProvisioningService`'s public signature. Confirm at merge time that:
- `lib/tasks/better_together/provision_tenant.rake` was updated consistently (it was, per the PR's own diff, reviewed as part of this assessment).
- No other caller of the service outside the PR's own changes references the old `admin:` keyword or assumes `'UTC'`/`'public'` defaults.
- `docs/production/multi_platform_deployment.md` (updated by this assessment) gets its "pending breaking change" callout removed/updated once merged.

*Effort estimate:* small — a single grep-and-verify pass at merge time, plus a one-line doc edit.
